# Phase 21 Design

**Date:** 2026-03-07
**Phase:** 21
**Source Specs:** `docs/phases/phase21.md`, `docs/dev/phase21.md`

## Goal

Deliver phase 21 end-to-end by adding a functional user menu in the dashboard header, a dedicated `/profile` experience for personal information and password changes, backend profile APIs including avatar upload, and the required automated coverage.

## Existing Context

- Backend is a Laravel 12 API already using Sanctum, JSON API responses, form requests, Pest feature tests, and file uploads on named disks.
- Frontend is a Next.js app-router application with Zustand stores, shadcn/radix primitives, Sonner toasts, and Vitest plus Playwright coverage.
- The current product already exposes a settings profile page under `/settings/profile`, a header user icon without behavior, and an auth store that persists the logged-in user.

## Design Decisions

### 1. Backend Profile Surface

- Add `avatar_path` persistence on `users` through a dedicated migration even if the model already anticipates the field, so the database contract matches the spec.
- Introduce a dedicated `ProfileController` under `Api\V1` with:
  - `GET /api/v1/profile`
  - `PATCH /api/v1/profile`
  - `POST /api/v1/profile/password`
- Keep validation in dedicated form requests:
  - `UpdateProfileRequest`
  - `UpdatePasswordRequest`
- Continue returning the authenticated user in the existing `{ status, message, data }` API envelope.

### 2. Avatar Storage

- Store avatars on the `public` disk under `avatars/{user-id}.{ext}` so the frontend can use a stable URL generated by Laravel storage.
- Expose `avatar_url` as an appended accessor on `User`.
- Replace the previous avatar file when uploading a new one and clear it when explicitly removed.

### 3. Password Change Semantics

- Require `current_password`, `password`, and `password_confirmation`.
- Validate the current password in the request layer and re-hash through Eloquent casting.
- Revoke all Sanctum tokens except the current one so the current session stays usable while other sessions are invalidated, matching the spec wording about "autres sessions".

### 4. Frontend State and UX

- Add a dedicated `profile` Zustand store for profile fetch/update/password actions while keeping the canonical authenticated user mirrored into `useAuthStore`.
- Replace the header ghost user button with a dropdown menu showing:
  - avatar or initials trigger
  - "Mon profil" navigation to `/profile`
  - logout action
- Build `/profile` as a dashboard page with two independent cards:
  - personal information and avatar upload
  - password change
- Use optimistic local previews for avatar selection and Sonner toasts for outcome feedback.

### 5. Testing Strategy

- Backend TDD first with dedicated feature coverage for get/update/password.
- Frontend component/store tests next for avatar upload and header menu behavior.
- Playwright last for the two required end-to-end flows, reusing existing authenticated dashboard patterns.

## Trade-offs

- A new `/profile` page is added instead of extending `/settings/profile` because the phase explicitly calls for a first-class profile page and a user-menu entrypoint.
- The auth store remains the single source for the current user object, while the profile store owns async actions, to avoid duplicating authentication concerns.
- Avatar deletion is implemented as an explicit boolean flag even though the spec only mentions upload; it simplifies replacing/removing previews and keeps the API practical without broadening scope much.

## Risks

- Existing middleware currently applies `two-factor` to protected routes, so the new endpoints must fit that contract without weakening auth behavior.
- Header menu behavior depends on client-side auth hydration; the trigger must remain stable before rehydration.
- Password token revocation must avoid deleting the current access token or the active browser session would log out immediately and break the expected flow.
